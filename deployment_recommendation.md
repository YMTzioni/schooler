# המלצת פריסה ואסטרטגיה טכנולוגית: Schooler Course Studio

**נכתב על ידי:** Manus AI
**תאריך:** יולי 2026

מסמך זה מציג ניתוח מעמיק והמלצות טכנולוגיות להעלאת פרויקט Schooler Course Studio מסביבת פיתוח מקומית לשירות אונליין יציב ואמין. ההמלצות מבוססות על הצרכים העסקיים, המגבלות הטכניות (כגון חסימות IP של יוטיוב וניהול סשנים), ועלויות התפעול.

---

## 1. ניתוח המצב הקיים והאתגרים המרכזיים

הפרויקט בנוי כיום כשרת Node.js/Express המשמש כ-BFF (Backend For Frontend) מול אפליקציית React. האתגרים המרכזיים במעבר לאונליין הם:

1.  **ניהול סשנים (State):** כרגע הסשנים נשמרים בזיכרון השרת (Map). בפריסה לענן (במיוחד Serverless כמו Vercel), הסשנים אובדים בכל cold start או פריסה מחדש, מה שמחייב את המשתמש להתחבר שוב ושוב [1].
2.  **חסימות IP של YouTube:** יוטיוב חוסמת בקשות לכתוביות המגיעות מכתובות IP של שרתי ענן (Datacenter IPs). הפתרון הנוכחי (פרוקסי חינמי) אינו יציב מספיק לסביבת פרודקשן [2].
3.  **מגבלות זמן ריצה (Timeouts):** הורדת כתוביות לפלייליסט שלם עשויה לארוך זמן רב ולחרוג ממגבלת ה-60 שניות של Vercel Serverless.
4.  **מודל משתמשים:** המערכת פועלת כיום ככלי אישי ללא מנגנון רישום או הפרדת נתונים בין משתמשים (Multi-tenant).

## 2. חלופות פריסה ואחסון

### חלופה א': פריסה מנוהלת (PaaS) - מומלצת ל-SaaS עתידי

חלופה זו משתמשת בשירותים מנוהלים לחלוטין, מה שמאפשר התמקדות בפיתוח במקום בתחזוקת שרתים.

*   **Frontend:** Vercel או Render (Static Site). שניהם מספקים CDN גלובלי ופריסה אוטומטית.
*   **Backend (API):** Render (Web Service) או Railway. שניהם מציעים שרת Node.js פועל תמיד (Always-on), מה שפותר את בעיית ה-cold starts של Vercel Serverless ומאפשר תהליכי רקע ארוכים (כמו הורדת כתוביות לפלייליסט) ללא מגבלת 60 שניות [1].
*   **ניהול סשנים:** Upstash Redis (Serverless Redis). שירות חינמי עד 10,000 בקשות ביום, מושלם לשמירת טוקנים של Schooler ו-Responder בצורה עמידה מחוץ לזיכרון השרת [3].

| רכיב | שירות מומלץ | עלות משוערת |
| :--- | :--- | :--- |
| Frontend | Vercel (Hobby) | $0 |
| Backend | Render (Starter) או Railway | ~$7/חודש |
| Session Store | Upstash Redis | $0 |
| **סה"כ** | | **~$7/חודש** |

### חלופה ב': שרת פרטי (VPS) - מומלצת לכלי אישי/ארגוני סגור

אם המערכת מיועדת לשימוש פנימי בלבד או למספר מצומצם של משתמשים, פריסה על שרת וירטואלי (VPS) באמצעות Docker היא הפתרון המשתלם והגמיש ביותר.

*   **תשתית:** Hetzner Cloud (שרת CX22 או CAX11).
*   **אריזה:** Docker Compose להרצת ה-Frontend (Nginx), ה-Backend (Node.js), ו-Redis (עבור סשנים) על אותו שרת [4].
*   **ניתוב ואבטחה:** Nginx Reverse Proxy עם תעודת SSL חינמית של Let's Encrypt (Certbot), או לחלופין Cloudflare Tunnel כדי לחשוף את השרת ללא פתיחת פורטים.

| רכיב | שירות מומלץ | עלות משוערת |
| :--- | :--- | :--- |
| VPS | Hetzner Cloud | ~$4.50/חודש |
| Session Store | Redis (בתוך Docker) | $0 |
| **סה"כ** | | **~$4.50/חודש** |

## 3. פתרונות לכתוביות יוטיוב (YouTube Subtitles)

כאמור, יוטיוב חוסמת באופן אקטיבי בקשות משרתי ענן. שימוש בפרוקסי חינמי אינו ישים לפרודקשן.

### גישה 1: שירות API מנוהל (מומלץ)

במקום לנהל פרוקסי ולשבור את הראש מול שינויים ביוטיוב, מומלץ להשתמש בשירות API ייעודי לכתוביות.

*   **Supadata API:** שירות המספק API נוח לשליפת כתוביות מיוטיוב. היתרון הגדול הוא מנגנון Fallback מבוסס AI (Whisper) במקרה שאין כתוביות מובנות. עלות: $5 ל-300 בקשות בחודש, או $17 ל-3,000 בקשות [5].
*   **YouTube-transcript.io:** שירות פשוט וזול יותר, המתמקד אך ורק בכתוביות יוטיוב ($9.99 ל-1,000 בקשות) [5].

### גישה 2: פרוקסי תושב (Residential Proxy)

שימוש בשירות כמו Bright Data או Decodo לניתוב בקשות ה-`yt-dlp` או `youtube-transcript-api` דרך כתובות IP של משתמשים ביתיים [2].
*   **יתרונות:** שליטה מלאה בקוד.
*   **חסרונות:** עלות משתנה (תשלום לפי תעבורה, לרוב סביב $3.50-$5.00 ל-GB), דורש תחזוקה.

**המלצה:** להתחיל עם שירות API מנוהל כמו Supadata. הוא חוסך זמן פיתוח ותחזוקה של תשתיות פרוקסי, ומספק תוצאות אמינות יותר בעלות צפויה.

## 4. ארכיטקטורה מומלצת לשלב ה-MVP

עבור השלב הראשון (MVP אונליין יציב), אני ממליץ על הארכיטקטורה הבאה:

1.  **Backend & Frontend:** פריסה על **Render** (חלופה א'). זה מאפשר לשרת ה-Node.js לרוץ ברצף (ללא מגבלת 60 שניות), ולשרת גם את קבצי ה-React הסטטיים (Monolith).
2.  **ניהול סשנים:** אינטגרציה עם **Upstash Redis**. יש להחליף את ה-`Map` בזיכרון בשרת לקריאה/כתיבה מ-Redis. זה יבטיח שהתחברויות לא יאבדו בעת פריסה מחדש.
3.  **כתוביות:** שימוש ב-**Supadata API** במקום קוד ה-scraping המקומי. זה יפתור לחלוטין את בעיית חסימות ה-IP.
4.  **תרגום:** המשך שימוש ב-Google Translate לא רשמי בשלב זה, אך תכנון מעבר ל-Google Cloud Translation API או DeepL ברגע שהתעבורה תגדל.

## 5. מענה לשאלות ממוקדות

### ארכיטקטורה ואחסון
*   **Vercel Serverless vs. שרת מלא:** שרת מלא (Render/Railway/Hetzner) עדיף משמעותית. הוא פותר את בעיית ה-timeout בכתוביות ארוכות ומונע אובדן סשנים תכוף.
*   **Redis:** חובה. Upstash היא בחירה מצוינת ללא עלות התחלתית. אין צורך ב-PostgreSQL בשלב זה אם אין ניהול משתמשים מורכב.
*   **Worker נפרד:** כרגע אין צורך. שרת מלא יכול לטפל בבקשות ארוכות. אם יעלה הצורך בעתיד, ניתן להוסיף תור (BullMQ).

### אבטחה
*   **שמירת סשנים:** מומלץ להצפין את ה-refresh tokens לפני שמירתם ב-Redis (למשל בעזרת `crypto` של Node.js), למרות ש-Upstash מאובטח.
*   **אימות משתמשים:** אם המערכת מיועדת לשימוש ארגוני, `.env` מספיק. אם היא תיפתח למשתמשים נוספים, חובה להוסיף מנגנון התחברות (למשל Auth0 או Clerk) מעל לאימות של Schooler.

### מודל עסקי
*   **Single-tenant vs. Multi-tenant:** המלצתי היא להתחיל ככלי אישי/ארגוני (Single-tenant). בניית SaaS מלא (Multi-tenant) דורשת מאמץ פיתוח משמעותי (הפרדת דאטה, חיובים, הרשאות) שעדיף לדחות עד להוכחת היתכנות (PMF) עם משתמשי קצה ראשונים.

## 6. סיכום ומפת דרכים לביצוע

1.  **שבוע 1:** פתיחת חשבון ב-Upstash Redis, עדכון קוד השרת לשמירת סשנים ב-Redis במקום בזיכרון.
2.  **שבוע 2:** הרשמה ל-Supadata API, עדכון מודול חילוץ הכתוביות לשימוש ב-API במקום ב-scraping מקומי.
3.  **שבוע 3:** פריסת הפרויקט (Frontend + Backend) ל-Render (או Hetzner VPS אם מעדיפים שרת פרטי).
4.  **שבוע 4:** בדיקות QA מקיפות אונליין, במיוחד על חילוץ פלייליסטים ארוכים ורענון טוקנים.

אסטרטגיה זו תספק סביבת פרודקשן יציבה, אמינה, ובעלות חודשית מינימלית, ותהווה בסיס איתן להרחבת המערכת בעתיד.
